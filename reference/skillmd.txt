````md
# Zipline Ops Platform

## Purpose

Use this skill to help build, review, or improve a production-grade operations platform based on the Zipline Command Center prototype.

The original prototype is a single-page HTML/JavaScript dashboard with in-memory mock data for:

- Order List / booking history
- Personnel database
- Transport assignment
- Pickup recheck / live feed
- Driver job sheets
- Staffing assignment
- Staff board
- Staff KPI
- Master operations log
- Pivot analysis
- Product/package database
- Excel export simulation

The production version must replace mock browser data with real backend services, persistent database storage, authentication, authorization, auditability, observability, automated testing, deployment pipelines, and scalable data workflows.

## Core Product Goal

Build a fast, beautiful, reliable internal operations platform for a zipline/tour activity business.

The platform must help operations teams:

1. Create and manage bookings.
2. Assign drivers to pickup jobs.
3. Track pickup status in near real time.
4. Move No Show customers to later rounds.
5. Assign field staff/guides to active guests.
6. Generate printable job sheets.
7. Track staff workload and KPIs.
8. Analyze bookings by agent, package, date, round, pax, join count, driver, and staff.
9. Export operational reports.
10. Maintain personnel and product/package master data.

## Recommended Tech Stack

### Frontend

Use:

- Next.js App Router
- TypeScript
- React Server Components where appropriate
- Tailwind CSS
- shadcn/ui
- TanStack Query for client-side server state when needed
- TanStack Table for large tables
- React Hook Form + Zod for forms
- Recharts or Tremor for charts
- date-fns or Day.js for date handling
- Playwright for E2E tests

Prioritize:

- Fast page loads
- Clean responsive UI
- Keyboard-friendly admin workflows
- Clear empty/error/loading states
- Printable A4 job sheets
- Virtualized tables for large datasets

Use virtualization for large lists with:

- `@tanstack/react-virtual`
- server-side pagination
- cursor pagination for large operational logs

### Backend

Prefer one of these production backend options.

#### Recommended default: NestJS + PostgreSQL

Use when the project is expected to grow into a serious multi-role operations system.

Use:

- NestJS
- PostgreSQL
- Prisma ORM or Drizzle ORM
- Redis for caching, queues, rate limiting, and realtime coordination
- BullMQ for background jobs
- WebSocket gateway or SSE for live operational updates
- OpenAPI/Swagger for API documentation

Why:

- Strong modular architecture
- Clean separation of domain modules
- Good for team-based backend development
- Scales better than ad hoc API routes
- Easier to enforce validation, auth, audit logs, and tests

#### Faster MVP option: Next.js full-stack + PostgreSQL

Use when speed matters more than long-term backend separation.

Use:

- Next.js Server Actions or Route Handlers
- PostgreSQL
- Prisma or Drizzle
- Redis only when needed
- NextAuth/Auth.js or Clerk

Avoid this option if backend complexity grows beyond simple CRUD and reporting.

#### High-scale option: Next.js + Go backend + PostgreSQL

Use only when there are very high throughput requirements, strict performance needs, or heavy concurrent realtime workloads.

Use:

- Next.js frontend
- Go backend with Fiber, Gin, or Chi
- PostgreSQL
- Redis
- NATS or Kafka if event volume becomes large

Do not choose Go just because it is fast. Choose it only when the team can maintain it.
````

Structure the system around domain modules, not pages.

* Create booking

* Search bookings

* Track pax and join counts
* Maintain booking lifecycle status

* `WAITING`
* `BOARDED`
* `NO_SHOW`
* `CANCELLED`
* `RESCHEDULED`

Important fields:

* booking number

* agent
* package
* customer name
* phone
* hotel
* room
* join count
* visitor count
* total pickup pax
* admin note
* source channel

### Transport Module

Responsibilities:

* Assign drivers
* View unassigned pickup jobs
* Filter by date and round
* Recheck pickup status
* Move No Show bookings to a later time slot
* Generate driver job sheets
* Track driver workload
* Print/export pickup job sheets

Rules:

* One booking may have one active driver assignment per pickup round.
* Reassignments must be auditable.
* No Show moves must record the previous round and reason.
* Driver job sheets must be printable and exportable.

### Staffing Module

Responsibilities:

* Assign field staff/guides
* Ensure minimum staff count per booking or package
* Display readiness status
* Show whiteboard-style staff board
* Calculate staff workload and KPI

Rules:

* Default minimum staff per active booking: 2.
* No Show or Cancelled bookings must not require staff assignment.
* Staff assignment changes must be auditable.
* KPI must exclude No Show and Cancelled bookings unless explicitly requested.

### Personnel Module

Responsibilities:

* Manage employees
* Define roles
* Track active/inactive status
* Support staff and driver role assignment
* Prevent deleting employees used in historical records

Roles:

* Operations Manager
* Transport Coordinator
* Staffing Coordinator
* Driver
* Field Staff / Guide
* Read-only Viewer

### Product / Package Module

Responsibilities:

* Maintain package database

* Define default staffing requirements
* Define whether package requires transport
* Define visibility and active/inactive state

Example packages:

* Extreme
* Express
* Exciting
* Gold Zipline
* Silver Zipline
* Extra
* Fast Track
* Fast Track + Luge
* Luge

### Reporting / Analytics Module

Responsibilities:

* Master operations log
* Pivot analysis by agent
* Pivot analysis by package
* Staff KPI
* Driver workload
* Daily capacity by round
* Export CSV/XLSX/PDF
* Scheduled report generation if requested

Rules:

* Large reports must run as background jobs.
* Reports must be filtered server-side.
* Exports must be permission-controlled.
* Generated files should expire after a configured retention period.

### Performance Indexes

Add indexes early for:

```sql
bookings(date, time_slot_id)

bookings(agent_id, date)

pickup_status_events(booking_id, created_at)
```

Use full-text search or trigram search for:

* customer name
* booking number

* hotel name

For PostgreSQL, consider:

* `pg_trgm`
* GIN indexes
* materialized views for heavy analytics
* partitioning by booking date if volume becomes very large

## Performance Standards

Target:

* Dashboard initial load under 2 seconds on normal office internet.
* Table interactions under 300ms for filtered/paginated data.
* API p95 latency under 500ms for common operations.
* Report generation moved to background job if over 3 seconds.
* Large tables must use server pagination or virtualization.
* Avoid loading all bookings into the browser.

Backend rules:

* Never fetch all historical bookings for normal views.
* Always filter by date range for operational views.
* Use cursor pagination for logs.
* Add database indexes before scaling data volume.
* Cache expensive aggregate reports.
* Use materialized views for heavy analytics if needed.

## Data Integrity Rules

Enforce these rules server-side:

1. Booking number should be unique per agent or globally, based on business rules.
2. Join count cannot be negative.
3. Visitor count cannot be negative.
4. Total pax equals join count plus visitor count.
5. Inactive drivers cannot be assigned to new transport jobs.
6. Inactive staff cannot be assigned to new field jobs.
7. Cancelled bookings cannot be boarded.
8. No Show move must create a status history record.
9. Staff KPI must use finalized boarded bookings by default.
10. Exported reports must match applied filters.

## Security Permission Matrix

Start with this RBAC model.

```text
Admin:
  full access

Operations Manager:
  bookings, transport, staffing, reports, personnel read

Transport Coordinator:
  bookings read, transport write, job sheets, pickup status

Staffing Coordinator:
  bookings read, staffing write, staff board, staff KPI

Driver:
  assigned job sheets only, pickup status update if mobile driver app exists

Field Staff:
  assigned work view only, no customer export by default

Viewer:
  read-only dashboards and reports
```

Always enforce permissions on the backend.

## Decision Guidance

When asked “what backend should we use?”, recommend:

```text
Use Next.js for the frontend and NestJS + PostgreSQL for the backend.

This gives a fast modern UI while keeping backend business logic, permissions, reporting, realtime events, audit logs, and large-data workflows cleanly separated. Use Redis for queues/cache/realtime coordination when data and operations grow.
```

If the user wants fastest MVP:

```text
Use Next.js full-stack with PostgreSQL first, but keep domain boundaries clean so the backend can be extracted to NestJS later.
```

If the user wants maximum long-term scale:

```text
Use Next.js frontend, NestJS or Go backend, PostgreSQL, Redis, background jobs, and materialized reporting tables.
```

## Anti-Patterns

Avoid:

* keeping all data in browser state
* using localStorage as production database
* one giant React component
* one giant API route
* exporting huge reports synchronously
* no audit logs
* no server-side permission checks
* no indexes on date/status filters
* loading all bookings into client memory
* mixing UI rendering with business rules
* relying on mock data after MVP planning
* making driver/staff workflows desktop-only if field usage is required
